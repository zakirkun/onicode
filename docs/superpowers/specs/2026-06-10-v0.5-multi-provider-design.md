# v0.5 Multi-Provider Support — OpenAI + Ollama

**Date:** 2026-06-10
**Status:** Approved

## Goal

Add OpenAI and Ollama as LLM providers alongside Anthropic, enabling OniCode to work with multiple model ecosystems while maintaining the existing agent architecture.

## Architecture

**Shared adapter pattern:** Both OpenAI and Ollama use the `openai` npm SDK since both expose OpenAI-compatible chat completions APIs. Ollama points at `http://localhost:11434/v1` via `baseUrl` config. One mapper, one provider class, config-driven difference.

**Why shared:** Ollama's OpenAI-compatible endpoint handles 95% of the API surface. Duplicating HTTP streaming, SSE parsing, and abort signal wiring would add maintenance burden with minimal benefit. Ollama-specific quirks (no `stream_options`, fake API key) handled in config, not code.

## Components

### 1. Dependencies

**Add:** `openai` npm package (v4.x, ESM-compatible).

```json
{
  "dependencies": {
    "openai": "^4.0.0"
  }
}
```

### 2. OpenAI Provider (`src/providers/openai/`)

**`mapper.ts`** — Pure conversion functions:
- `toOpenAIMessages(messages: ChatMessage[]): ChatCompletionMessageParam[]`
  - Maps canonical `ChatMessage[]` to OpenAI format
  - Role mapping: `user` → `user`, `assistant` → `assistant`, `tool` → `tool`
  - Tool results: `{ role: "tool", tool_call_id, content }`
  - System messages dropped (extracted separately as `system` param)
- `toOpenAITools(tools: ToolDefinition[]): ChatCompletionTool[]`
  - Maps canonical tool defs to OpenAI function calling format
  - `{ type: "function", function: { name, description, parameters } }`
- `mapOpenAIStopReason(reason: string | null): StopReason`
  - `"stop"` → `"end_turn"`
  - `"tool_calls"` → `"tool_use"`
  - `"length"` → `"max_tokens"`
  - `"content_filter"` → `"stop_sequence"`
  - `null` / other → `"error"`

**`provider.ts`** — `OpenAIProvider implements LLMProvider`:
- Constructor: creates `OpenAI` SDK client with `apiKey` + `baseUrl`
- `stream(req, signal)`:
  - Calls `client.chat.completions.create({ stream: true, ... })`
  - Iterates SSE chunks, buffers tool-call arguments by index
  - Yields `ChatChunk` events:
    - `delta.content` → `{ kind: "text", delta }`
    - `delta.tool_calls[i].function.arguments` → accumulate in `ToolCallBuffer[index]`
    - `finish_reason` → flush tool_calls as `{ kind: "tool_call", ... }`, then `{ kind: "stop", reason, usage }`
  - Abort signal wired via `stream.controller.abort()`
- `countTokens(messages)`:
  - Uses `client.chat.completions.create({ max_tokens: 1, ... })` to get token counts from response headers
  - Falls back to heuristic (4 chars ≈ 1 token) if unavailable

**Tool-call buffering:**
```typescript
interface ToolCallBuffer {
  [index: number]: {
    id: string;
    name: string;
    arguments: string; // accumulated JSON fragments
  };
}
```

### 3. Provider Factory (`src/providers/registry.ts`)

Update `createProvider()` switch cases:

```typescript
case "openai":
  return new OpenAIProvider({
    apiKey: readApiKey(config, "OPENAI_API_KEY"),
    baseUrl: config.baseUrl, // optional
  }, log);

case "ollama":
  return new OpenAIProvider({
    apiKey: "ollama", // Ollama ignores this but SDK requires a value
    baseUrl: config.baseUrl ?? "http://localhost:11434/v1",
  }, log);
```

Both construct `OpenAIProvider` with different config. No code duplication.

### 4. Ollama-Specific Handling

**Config differences (not code):**
- `baseUrl: "http://localhost:11434/v1"` (note `/v1` suffix — Ollama's OpenAI compat lives there)
- `apiKey: "ollama"` (fake key; Ollama ignores but SDK requires)
- No `stream_options` sent (Ollama rejects unknown fields)

**Implementation:** `OpenAIProvider.stream()` checks `this.id === "ollama"` to skip `stream_options: { include_usage: true }`. Clean, localized, no abstraction leak.

### 5. Error Handling

**SDK errors → typed exceptions:**
- `APIError` (401, 403, 429, 500) → catch and rethrow with context
- Network errors → throw with message
- Abort → clean exit, no error

**Example:**
```typescript
try {
  const stream = await this.client.chat.completions.create(params);
  // ... iterate
} catch (err) {
  if (err instanceof APIError) {
    throw new Error(`OpenAI API error ${err.status}: ${err.message}`);
  }
  throw err;
}
```

## Data Flow

```
User message →
  Agent →
    LLMProvider.stream(req, signal) →
      OpenAIProvider →
        openai SDK →
          OpenAI API / Ollama API →
        SSE chunks →
          toOpenAIMessages() / toOpenAITools() (outbound) →
          ChatChunk yield (inbound) →
            text delta →
            tool_call (buffered) →
            stop →
      Agent processes chunks →
        tool_call → execute tool → recurse
        text → stream to user
        stop → end turn
```

## Success Criteria

1. `onicode run -p "Hello" --provider openai --model gpt-4o` works end-to-end
2. `onicode run -p "Hello" --provider ollama --model llama3.1` works with local Ollama
3. Tool calling works for both (function calling mapped)
4. Streaming works (no buffering entire response)
5. Abort signal works (Ctrl+C cancels in-flight)
6. All tests pass: `pnpm test`
7. Type check clean: `pnpm typecheck`
8. No `any` violations

## Out of Scope

- Non-streaming mode (streaming only, matches Anthropic)
- Vision/multimodal (text + tools only)
- Embeddings, image generation, audio
- Provider-specific features (OpenAI response format, Ollama keep-alive)
- Model capability detection (user must know their model supports tools)

## Risks & Mitigations

**Risk:** Ollama tool support varies by model (llama3.1+, mistral support it; others don't).
**Mitigation:** Adapter sends tools regardless; model handles or ignores. Document in README that not all Ollama models support tools.

**Risk:** `openai` SDK adds dependency.
**Mitigation:** Well-maintained, ESM-compatible, no transitive bloat. Worth the maintenance savings.

**Risk:** Ollama rejects `stream_options`.
**Mitigation:** Localized check in `OpenAIProvider.stream()`. No abstraction leak.

## Conclusion

v0.5 adds OpenAI + Ollama support with minimal code via shared `openai` SDK adapter. One mapper, one provider class, config-driven difference. Maintains OniCode's pure-core philosophy (mapper is pure, provider handles I/O).
